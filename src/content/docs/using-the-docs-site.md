# Using the Docs Site

This site is our shared working memory for projects we build, research, and operate together.

## What to document here

Use the docs site for things that future Shaun or future Amiga should be able to recover quickly.

That includes:
- project summaries
- setup instructions
- architecture notes
- workflows and routines
- decision logs
- recurring research conclusions
- business operating patterns
- pitfalls and fixes

## What not to document here

Do not put everything here.

This is not for:
- raw scratch notes
- temporary chat fragments
- secrets
- one-off throwaway ideas unless they turn into something reusable

## Good prompts to use with me

- Document this in the docs site.
- Create a project page for this.
- Turn this into a decision log.
- Write setup docs for this workflow.
- Add a troubleshooting note for this fix.
- Summarize this project into docs.
- Convert this discussion into a reusable playbook.

## Recommended structure

### Home
The front door and index.

### Operating System
Shared workflows, routines, patterns, and ways of working.

### Projects
One page per real project.

### Decisions
Short durable decision records.

### Playbooks
Repeatable procedures and routines.

### Setup
How tools, services, accounts, and integrations work.

## Lightweight documentation rule

If we had to rediscover this in two weeks, document it.

If it would cost more than five minutes to reconstruct, document it.

If it affects how we work more than once, document it.

## Suggested writing style

- concise
- factual
- practical
- no fluff
- enough context to recover quickly

## Default page types we should create often

- Project page
- Decision page
- Setup page
- Playbook page
- Troubleshooting page

## Long-term goal

The docs site should become the place where:
- project memory survives
- workflows improve over time
- setup knowledge stops disappearing
- operating patterns become reusable
